# Инструкция: аудит покупки DMVSOS

Философия: мы не продаём «горячо» и не давим. Берём честностью, открытостью и простым,
интуитивным дизайном оплаты. Каждый пункт ниже проверяет одно из трёх.
Прогонять: после любого изменения цен/checkout/upgrade и раз в месяц.

## Часть 1. Данные: что случилось на самом деле

Все команды выполняются с live-ключом (`vercel env pull --environment=production`,
переменная `STRIPE_SECRET_KEY`). Ключ не коммитить, файл держать вне репо.

### 1.1 Все checkout-сессии за 30 дней

```bash
curl -s -u "$STRIPE_SECRET_KEY:" -G https://api.stripe.com/v1/checkout/sessions \
  -d limit=50 -d "created[gte]=$(date -v-30d +%s)"
```

Смотреть на каждую сессию:
- `status` / `payment_status`: `complete/paid` = продажа, `expired/unpaid` = бросил.
- `customer_email`: кто (после login-before-checkout email всегда есть = можно понять юзера).
- `metadata.pass_type` / `kind`: что покупал. Сессии без metadata = не из приложения (платёжные ссылки владельца), в воронку не считать.
- Повторные сессии одного email за минуты = юзер ХОТЕЛ купить и упёрся в стену. Самый важный сигнал.

### 1.2 Дошёл ли человек до ввода карты

```bash
curl -s -u "$STRIPE_SECRET_KEY:" https://api.stripe.com/v1/checkout/sessions/<cs_live_...>
# поле payment_intent:
#   null                → карту даже не начал вводить: проблема СТРАНИЦЫ (доверие/ясность/методы)
#   pi_... есть         → тянуть PI и читать last_payment_error:
curl -s -u "$STRIPE_SECRET_KEY:" https://api.stripe.com/v1/payment_intents/<pi_...>
#   decline_code = insufficient_funds / do_not_honor / ... → проблема КАРТЫ, не наша
```

Это главная развилка диагноза: страница отпугнула или банк отказал. Лечатся по-разному.

### 1.3 Сверка с GA4

`begin_checkout` (GA4, с 2026-06-29) должен сходиться 1:1 с числом созданных сессий
в Stripe. Расхождение = create-checkout падает или трекинг сломан.
`purchase` должен сходиться с `complete/paid`.

### 1.4 Бенчмарк

Hosted Stripe Checkout для one-time digital: норма завершения 40-60%.
Ниже 25% = страница что-то делает не так. Считать: paid / уникальные email с сессиями.

## Часть 2. Глазами покупателя: чек-лист страницы оплаты

Создать аудит-сессию (не платить, только смотреть; live-сессия истекает через 24 ч):

```bash
curl -s -u "$STRIPE_SECRET_KEY:" https://api.stripe.com/v1/checkout/sessions \
  -d mode=payment -d "line_items[0][price]=<price_id>" -d "line_items[0][quantity]=1" \
  -d "success_url=https://dmvsos.com/success?session_id={CHECKOUT_SESSION_ID}" \
  -d "cancel_url=https://dmvsos.com/upgrade" -d "metadata[audit]=internal_checkout_review"
```

Открыть URL на десктопе И на телефоне. Проверить каждый пункт:

### Честность (обещание на странице оплаты = обязательство, которое мы выполняем)
- [ ] Название и описание продукта обещают только то, что мы реально делаем.
      Pass Guarantee у CDL Pro - НАСТОЯЩЕЕ обязательство (владелец лично доводит
      каждого оплатившего до сдачи, в этом философия проекта). Оно остаётся.
      Проверка здесь: не появилось ли обещание, которое мы выполнять не готовы.
- [ ] «One-time payment / no subscription» видно ДО кнопки Pay.
- [ ] Цена на странице = цена, которую юзер видел на сайте. Никаких доплат-сюрпризов.
- [ ] Статус-дескриптор (`DMVSOS.COM`) совпадает с тем, где юзер покупал: не будет «что это за списание?»

### Доверие
- [ ] Логотип и фирменный цвет загружены (Stripe Dashboard → Settings → Branding). Безликая страница = скам-сигнал.
- [ ] Имя бизнеса написано по-брендовому (не lowercase-огрызок).
- [ ] Support email указан (Settings → Business details) и реально читается.
- [ ] У продукта есть картинка: левая половина страницы не должна быть пустой серой дырой.

### Простота
- [ ] Список методов оплаты короткий и уместный. Красные флаги для нашей аудитории: Crypto,
      Klarna-рассрочка на $29.99, «$5 back» банк-промо. Каждый лишний метод = +1 решение и −доверие.
- [ ] Apple Pay / Google Pay работают (60% трафика mobile): открыть с iPhone, кнопка должна быть первой.
- [ ] Email предзаполнен (login-before-checkout) - юзер не вводит его руками.
- [ ] Ноль обязательных полей сверх карты (телефон/адрес НЕ спрашиваем).
- [ ] Язык страницы = язык юзера (`locale` из `body.lang`; проверить сессии в Stripe: у ru-юзеров locale=ru).

### После оплаты
- [ ] /success открывается, доступ выдан мгновенно (профиль is_pro/active_passes), GA4 purchase ушёл.
- [ ] Письмо-чек Stripe приходит и выглядит прилично.
- [ ] Отмена (кнопка «назад») возвращает на /upgrade без потери выбора.

## Часть 3. Снимок аудита 2026-07-08 (baseline)

Ни один из бросивших не начал вводить карту → проблема страницы, не банков.
Двое (pakthanan ×3 за 11 мин, his_apple ×3 за 16 мин) возвращались трижды: хотели купить.

Найдено и что с этим стало:
1. Pass Guarantee у CDL Pro: решение владельца 2026-07-08 - остаётся. Это реальное
   обязательство (каждого оплатившего лично доводим до сдачи), а не маркетинг.
2. СДЕЛАНО 2026-07-08: имена продуктов без длинных тире («Auto Pass (30 days)» и т.д.),
   у всех 4 продуктов добавлены картинки (vehicles/*-hero.png, key.png) - левая
   половина checkout больше не пустая.
3. СДЕЛАНО 2026-07-08 (код create-checkout): `submit_type: 'pay'` + локализованная
   строка под кнопкой Pay «Разовый платёж. Доступ на 30 дней. Это не подписка,
   отменять ничего не нужно» ×5 языков, только payment mode. Проверено живой сессией.
4. Методы оплаты: решение владельца - у человека должен быть выбор, НЕ выключаем

5. ОСТАЛОСЬ (только Dashboard, API на своём аккаунте запрещён Stripe):
   Settings → Branding: иконка + фирменный цвет #0B1C3D (сейчас страница безликая);
   Settings → Business details: имя «DMVSOS» (сейчас lowercase «dmvsos») +
   support email maindmvsos@gmail.com (он уже публичный на сайте).
6. Работает и НЕ трогать: locale по языку юзера, предзаполненный email, дубль-гард 409,
   Apple Pay домен активен, вебхук выдаёт доступ (проверено на покупке 2026-06-19).
